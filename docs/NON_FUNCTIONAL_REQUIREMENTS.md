## 2. 가용성 요구사항

### 2.1 서비스 가동률 (SLA)

| 서비스 | 목표 가동률 | 허용 월간 다운타임 |
|--------|------------|------------------|
| Gateway-Service | 99.9% | 43분 |
| User-Service (인증) | 99.9% | 43분 |
| Workspace-Service | 99.5% | 3시간 39분 |
| Ingestion-Service | 99.0% | 7시간 18분 |
| Commerce-Service | 99.9% | 43분 |
| Intelligence-Service (AI) | 99.0% | 7시간 18분 |
| 프론트엔드 (brainx-next) | 99.9% | 43분 |

### 2.2 장애 대응

- **서비스 격리**: 개별 마이크로서비스 장애가 전체 시스템으로 전파되지 않아야 한다
  - AI/Intelligence-Service 장애 시 노트 읽기/쓰기 기능은 정상 동작해야 한다
  - Ingestion-Service 장애 시 노트 에디터 기능은 영향받지 않아야 한다
- **자동 재시작**: Docker `restart: unless-stopped` 정책 적용 (현재 구성 유지)
- **DB 헬스체크**: PostgreSQL, Redis, Neo4j 모두 healthcheck 이후 서비스 기동
- **결제 장애 격리**: Commerce-Service 장애 시 기존 구독 사용자의 서비스 이용이 차단되지 않아야 한다

### 2.3 복구 목표

| 항목 | 목표 |
|------|------|
| RTO (Recovery Time Objective) | 30분 이내 |
| RPO (Recovery Point Objective) | 1시간 이내 데이터 손실 허용 |
| 드래프트 자동저장 주기 | 30초 이내 |
| 노트 버전 충돌 감지 | 저장 시 즉시 |

## 3. 확장성 요구사항

### 3.1 수평 확장

- 모든 백엔드 마이크로서비스는 **무상태(Stateless)** 로 설계하여 인스턴스 수평 확장이 가능해야 한다
- 세션 상태는 Redis에 저장하고 서비스 인스턴스 간 공유 가능해야 한다
- Gateway-Service는 라운드로빈 또는 최소 연결 기반 로드밸런싱을 지원해야 한다

### 3.2 데이터 확장

| 항목 | 초기 목표 | 확장 목표 |
|------|-----------|-----------|
| 사용자 수 | 1,000명 | 100,000명 |
| 사용자당 노트 수 | 500개 | 10,000개 |
| 노트 평균 크기 | 10KB | 100KB |
| 전체 저장 노드 (Neo4j) | 500,000개 | 5,000,000개 |
| 파일 저장소 | 100GB | 10TB |

### 3.3 AI 워크로드 확장

- Intelligence-Service는 LLM API(OpenAI, Claude 등) 호출 실패 시 재시도 및 Fallback 모델 전환을 지원해야 한다
- 시맨틱 검색을 위한 벡터 DB(Qdrant)는 샤딩(Sharding)을 통해 노드 수에 비례한 확장이 가능해야 한다
- RAG 파이프라인은 사용자별 독립적으로 실행되어 다른 사용자 요청의 지연이 없어야 한다

## 4. 보안 요구사항

### 4.1 인증 및 인가

| 항목 | 요구사항 |
|------|----------|
| 인증 방식 | JWT Access Token (Bearer) + Refresh Token (HttpOnly Secure Cookie) |
| Access Token 유효 시간 | 15분 이하 |
| Refresh Token 유효 시간 | 30일 이하 |
| 소셜 로그인 | OAuth 2.0 표준 준수 (Google, Kakao, Naver) |
| JWT 서명 비교 | 상수 시간 비교 (타이밍 공격 방지) |
| 비인증 요청 처리 | 401 또는 403 반환, dev fallback 금지 |
| 권한 없는 리소스 접근 | 403 반환 및 세션 클리어 |

### 4.2 데이터 보안

- **전송 암호화**: 모든 클라이언트-서버 통신은 HTTPS/TLS 1.2 이상 적용
- **저장 데이터 암호화**: 민감 정보(비밀번호, 소셜 토큰)는 BCrypt 또는 AES-256으로 암호화 저장
- **환경 변수 관리**: API Key, Secret, DB 비밀번호는 `.env` 파일 또는 시크릿 매니저를 통해 관리하며 코드에 하드코딩 금지
- **OAuth 상태 관리**: OAuth state 토큰은 10분 TTL 이후 자동 만료

### 4.3 입력 검증 및 XSS 방지

| 항목 | 요구사항 |
|------|----------|
| URL 입력 검증 | `http://`, `https://` 스킴만 허용, `javascript:` 스킴 차단 |
| 마크다운 HTML 렌더링 | `javascript:` URL 링크 생성 차단 |
| 백엔드 URL 저장 검증 | ExtensionCaptureService에서 URL 스킴 검증 후 저장 |
| SQL Injection | JPA/QueryDSL 파라미터 바인딩 사용, 동적 쿼리 직접 조합 금지 |
| 파일 업로드 | MIME 타입 및 파일 확장자 서버 측 검증, 허용 형식 외 거부 |

### 4.4 API 보안

- **Rate Limiting**: 로그인/회원가입 API는 IP당 분당 10회 이하 제한
- **CORS 정책**: `FRONTEND_BASE_URL` 환경 변수 기반 Origin 허용, `chrome-extension://*` 명시적 허용
- **Internal API 격리**: `/internal/v1/**` 경로는 서비스 간 통신 전용으로 외부 접근 차단
- **민감 정보 노출 차단**: 에러 응답에 스택 트레이스, DB 쿼리, 내부 서비스 URL 미포함

### 4.5 결제 보안

- Toss Payments 결제 승인은 반드시 서버 측 confirm API를 통해 처리 (클라이언트 단독 처리 금지)
- 결제 관련 웹훅 수신 시 서명 검증 필수

## 5. 데이터 요구사항

### 5.1 데이터 정합성

| 항목 | 요구사항 |
|------|----------|
| 노트 저장 | 낙관적 잠금(Optimistic Locking) 기반 버전 충돌 감지 |
| 서비스 간 데이터 동기화 | 이벤트 기반 전파, DB 직접 공유 금지 |
| 폴더 삭제 | 하위 노트 처리 정책 명시 (소프트 삭제 또는 계단식 삭제) |
| 공유 링크 만료 | 만료일 null 처리 방어 로직 및 기본값(30일) 부여 |

### 5.2 데이터 백업

| 항목 | 요구사항 |
|------|----------|
| PostgreSQL 백업 주기 | 매일 1회 전체 백업, 1시간마다 WAL 아카이빙 |
| Neo4j 백업 주기 | 매일 1회 전체 백업 |
| Redis 백업 | RDB 스냅샷 + AOF 영속성 설정 |
| 백업 보관 기간 | 30일 |
| 백업 복구 테스트 | 분기 1회 복구 시나리오 검증 |

### 5.3 데이터 보존 및 삭제

- 사용자 탈퇴 요청 시 30일 이내 개인 데이터 완전 삭제 처리
- 삭제된 노트는 소프트 삭제로 처리하며 30일간 복구 가능한 상태 유지
- 임포트 작업 로그는 90일 보관 후 자동 삭제

### 5.4 드래프트 및 버전 관리

- 노트 편집 중 드래프트는 Redis에 임시 저장 후 30초 주기로 영구 저장
- 드래프트 Redis TTL은 24시간으로 설정
- 노트 버전 이력은 최근 50개까지 보관

## 7. 운영성 요구사항

### 7.1 모니터링

| 항목 | 요구사항 |
|------|----------|

| 대시보드 | 서비스별 RPS, 레이턴시, 에러율을 실시간으로 확인 가능해야 함 |

### 7.2 로깅

| 항목 | 요구사항 |
|------|----------|
| 로그 형식 | JSON 구조화 로그 (타임스탬프, 서비스명, 요청 ID, 레벨, 메시지) |
| 요청 추적 | `traceId`를 모든 서비스 로그에 전파하여 분산 추적 가능 |
| 로그 레벨 | 운영 환경에서 INFO 이상, 민감 정보(토큰, 비밀번호) 마스킹 |
| 로그 보관 | 운영 로그 90일 보관 |
| 에러 로그 | 5xx 에러 발생 시 스택 트레이스 포함 로그 기록 |

### 7.3 알림 및 장애 대응

- 서비스 다운 감지 후 1분 이내 알림 발송
- 결제 실패율 5% 초과 시 즉시 알림
- AI 토큰 사용량이 일일 한도의 80% 도달 시 경고 알림
- Notion OAuth 연동 실패율 10% 초과 시 알림

| 데이터 처리 위탁 | 외부 API(OpenAI, Toss Payments 등) 위탁 처리 현황 문서화 |
| 열람/삭제 요청 | 사용자 개인정보 열람 및 삭제 요청을 30일 이내 처리 |
| 개인정보 이전 | 국외 서버 전송 시 해당 법령에 따른 조치 이행 |

### 9.2 결제 보안 (PCI-DSS 준수)

- 카드 번호, CVC 등 결제 민감 정보를 자체 서버에 저장하지 않음
- 결제 처리는 Toss Payments PG사에 위임
- 결제 승인은 반드시 서버 측 API 호출로 처리
