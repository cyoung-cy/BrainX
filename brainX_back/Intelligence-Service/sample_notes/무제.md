### ① Database per Service (데이터 격리)
* 각 서비스는 자신만의 데이터베이스(스키마)를 독점적으로 소유합니다.
* **절대 규칙:** 어떤 서비스도 다른 서비스의 테이블을 직접 조회(`JOIN`)하거나 수정할 수 없습니다. 데이터가 필요하다면 오직 API 호출이나 이벤트 구독을 통해서만 공유합니다. 가장 강력한 수준의 결합도 해제(Decoupling)를 달성합니다.

### ② 이벤트 기반 파생 데이터 (비동기 동기화)
* 노트 서비스(Core)에서 데이터 변경(생성/수정/삭제)이 발생하면, 즉시 `NoteCreated`, `NoteUpdated`, `NoteDeleted`와 같은 비동기 이벤트만 메시지 브러커로 발행합니다.
* 검색, 그래프, 분석 등의 주변 서비스는 이 이벤트를 각자 구독(Subscribe)하여 자체 데이터 저장소를 동기화합니다. 노트 서비스는 자신의 이벤트를 누가 쓰는지 전혀 알 필요가 없습니다.

### ③ 무상태(Stateless) JWT 인증 (인증 분산)
* 인증 서비스는 사용자가 로그인할 때 검증된 `JWT(JSON Web Token)`를 발급하는 역할만 수행합니다.
* 나머지 서비스나 API 게이트웨이는 전역 세션 저장소를 조회하지 않고, 전달받은 JWT의 서명을 각자 로컬에서 무상태로 검증하여 요청을 처리합니다. 매 요청마다 인증 서비스에 질의하는 병목 현상을 제거합니다.

### ④ 임베딩·LLM 중앙화 (AI 자원 격리)
* 텍스트의 벡터화(임베딩 생성)는 **검색 서비스**가 전담하고, 외부 AI 모델 호출 및 API 키 관리는 **LLM 게이트웨이**가 전담합니다.
* 다양한 서비스 내의 AI 응용 기능(챗봇, 요약, 편집 보조 등)들이 개별적으로 AI 라이브러리나 API 키를 중복 소유하지 않고, 중앙화된 두 고성능 서비스만을 호출하여 가볍게 유지됩니다.

## 🏗️ 3. 2단계: 백엔드 MSA 서비스 분해 (4개 티어 구조)

백엔드 시스템은 비즈니스 중요도, 변경 주기, 자원 사용 특성에 따라 총 **4개의 논리 티어(Tier), 14개의 마이크로서비스**로 격리 설계되었습니다.

## ⚡ 4. 핵심 충돌 지점의 아키텍처적 해결 방안

마이크로서비스 경계선 상에서 데이터 소유권이 모호해질 수 있는 충돌 지점을 명확한 규칙으로 해결했습니다.

1.  **AI 연결 제안 vs 최종 링크 저장 (`4.3.1 백링크`)**
    * **충돌:** AI 서비스가 문맥을 분석해 링크를 제안하고 사용자가 이를 수락할 때 데이터 처리 주체 문제.
    * **해결:** 링크 제안의 계산 및 추론은 **S7/S8(AI 계층)**이 수행하지만, 사용자가 최종 승인하여 확정된 정식 링크 데이터는 구조 메타데이터의 진실 공급원인 **S4(구조·분류 서비스)**에만 저장됩니다.
2.  **토큰 사용량 관리 vs 관리자 통계 화면 (`10.1 토큰 관리`)**
    * **충돌:** 사용자의 토큰 사용량 정보와 이를 모니터링해야 하는 운영자 대시보드 간의 데이터 중복 소유 문제.
    * **해결:** 실시간 토큰 사용 데이터의 집계 로직 및 원천 과금 데이터 소유권은 완전히 **S12(결제·토큰 서비스)**가 가집니다. **S14(관리자 BFF)**는 데이터를 복제하여 가지지 않고, 운영자 화면 요청이 들어올 때 S12의 통계 API를 원격 호출하여 표출만 함으로써 **데이터 중복 소유 및 동기화 오류를 원천 차단**했습니다.

# 🏛️ MSA 서비스 분해 및 아키텍처 원칙 정의서

본 문서는 시스템의 복잡도를 낮추고 지속 가능한 확장성을 확보하기 위해 수립된 서비스 분해 원칙(Decomposition Principles)과 이를 기반으로 설계된 **14개 마이크로서비스(MSA)의 아키텍처 명세 및 티어링 구조**를 다룹니다.

## 💡 1. 분해 원칙 (왜 이렇게 나누는가?)

서비스를 완전히 독립적이고 자율적인 컴포넌트로 격리하기 위해 다음 **4대 핵심 디커플링 장치**를 강제합니다.

- **Database per Service (데이터 격리):** 각 서비스는 자신만의 데이터베이스(스키마)를 독점적으로 소유합니다. 어떤 서비스도 다른 서비스의 테이블을 직접 조회(`JOIN`)하거나 수정할 수 없으며, 오직 API나 이벤트를 통해서만 데이터를 공유합니다. (가장 강력한 디커플링)
    
- **이벤트 기반 파생 데이터 (비동기 동기화):** 노트 서비스(Core)에서 데이터가 변경되면 즉시 `NoteCreated`, `NoteUpdated`, `NoteDeleted`와 같은 비동기 이벤트만 발행합니다. 검색, 그래프, 분석 등의 서비스는 이 이벤트를 비동기로 구독(Subscribe)하여 각자의 DB를 갱신합니다. 노트 서비스는 누가 이 데이터를 구독하는지 몰라도 됩니다.
    
- **무상태(Stateless) JWT 인증 (인증 분산):** 인증 서비스가 토큰을 발급하면, API 게이트웨이 및 개별 서비스는 토큰의 서명을 각자 로컬에서 검증만 합니다. 매 요청마다 인증 서비스에 질의하는 병목 현상을 제거합니다.
    
- **임베딩·LLM 중앙화 (AI 자원 격리):** 텍스트의 벡터화(임베딩 생성)는 **검색 서비스**, 외부 모델 호출 및 API 키 관리는 **LLM 게이트웨이**에 모아서 처리합니다. 다른 AI 기능들이 중복 구현 없이 중앙 서비스를 호출만 하도록 설계하여 아키텍처를 가볍게 유지합니다.

## ⚡ 4. 핵심 충돌 지점의 아키텍처적 해결 방안

마이크로서비스 경계선 상에서 데이터 소유권이 모호해질 수 있는 충돌 지점을 명확한 규칙으로 해결했습니다.

1. **AI 연결 제안 vs 최종 링크 저장 (`4.3.1 백링크`)**
    
    - **충돌:** AI 서비스가 문맥을 분석해 링크를 제안하고 사용자가 이를 수락할 때 데이터 처리 주체 문제.
        
    - **해결:** 링크 제안의 계산 및 추론은 S7/S8(AI 계층)이 수행하지만, 사용자가 최종 승인하여 확정된 정식 링크 데이터는 구조 메타데이터의 진실 공급원인 S4(구조·분류 서비스)에만 저장됩니다.
        
2. **토큰 사용량 관리 vs 관리자 통계 화면 (`10.1 토큰 관리`)**
    
    - **충돌:** 사용자의 토큰 사용량 데이터 소유권과 이를 조회해야 하는 관리자 화면 간의 데이터 중복 소유 문제.
        
    - **해결:** 실시간 토큰 사용 데이터의 집계 및 원천 과금 데이터 소유권은 완전히 S12(결제·토큰 서비스)가 가집니다. S14(관리자 BFF)는 데이터를 복제하지 않고, 운영자 화면 요청 시 S12의 통계 API를 원격 호출하여 표출만 함으로써 데이터 중복 소유를 피했습니다.
