# 주제 배경
#### 1. 검색·발견의 실패 (못 찾는 문제)
- **작성했지만 못 찾음**: 분명히 적어놓은 내용인데 어떤 폴더, 어떤 노트에 있는지 기억나지 않아 검색에 시간을 허비함
- **정확한 단어가 기억나지 않음**: "그때 그 개념"이 떠오르는데 정확한 키워드를 몰라서 검색이 실패함 → 키워드 검색의 근본적 한계
- **폴더 구조의 함정**: 분류 기준을 처음에 잘못 정하면, 나중에 어디에 넣었는지 본인도 헷갈림 (예: "AI"로 넣었나 "공부"로 넣었나)

#### 2. 중복·비효율의 문제 (이미 했는데 또 함)
- **중복 작성**: 같은 내용을 정리해놓고도 존재 자체를 몰라서 처음부터 다시 작성함 → 시간 낭비+ 일관성 깨짐
- **연결의 부재**: 서로 관련된 메모인데 따로따로 흩어져 있어서, 합쳐서 보면 새로운 인사이트가 될 것들이 그냥 죽은 텍스트로 남음
- **수동 링크의 피로감**: Obsidian처럼 `[[ ]]`로 직접 연결해야 하는 도구는, 결국 "정리하는 행위" 자체가 또 다른 일이 되어버림 (정리하다가 지침)

#### 3. 기억·체화의 실패 (적었지만 안 남음)
- **단순 기록 = 휘발성 기억**: 타이핑은 했지만 다시 보지 않으면 머릿속에 남지 않음 (그냥 "저장했다"는 안도감만 남음)
- **복습 트리거의 부재**: 예전에 적은 좋은 인사이트가 있어도, 다시 꺼내볼 계기가 없어서 그대로 묻힘
- **지식 간 연결의 부재 → 통찰 부재**: 점(점) 형태의 메모만 쌓이고, 점과 점을 잇는 "선"이 없어서 새로운 생각으로 발전하지 못함

#### 4. 기존 도구의 구조적 한계 (AI 도구 vs 노트 도구의 간극)
- **AI 도구(ChatGPT 등)**: 질문엔 답해주지만 "내 자료"를 기반으로 하지 않음 → 매번 컨텍스트를 새로 설명해야 함, 내 지식 누적이 안 됨
- **노트 도구(Notion, Obsidian)**: 저장은 잘하지만 정리는 결국 사람이 다 해야 함 → "정리해야 하는데..."라는 부담이 오히려 기록을 멀리하게 만듦
- **결과적으로 두 도구 사이에 비어 있는 영역**: "내 기록을 기반으로 AI가 알아서 정리·연결해주는" 도구가 없었음 → 이 공백이 BrainX의 존재 이유

# 주 기능 3가지
### 핵심 기능 3가지 (Core Features)

#### 1️⃣ AI 자동 연결 — 백링크 & 마인드맵 (3.5, 4.3, 6.1~6.3)

사용자가 노트를 작성하기만 하면, AI가 의미적으로 유사한 노트를 탐지해 연결을 제안하고 2D/3D 마인드맵으로 시각화한다.

- 수동 `[[ ]]` 링크가 아니라 **AI가 맥락 분석 후 "연결 제안"**
- 노드 클릭 시 3줄 요약 마우스오버, 클러스터링으로 주제별 그룹 자동 형성
- **발표 포인트**: "정리는 AI가, 당신은 생각만"의 핵심 구현체

#### 2️⃣ RAG 기반 지식 챗봇 (2.1.2, 5.1.1)

키워드 검색이 아니라 **내 노트를 근거로 답하는 AI 대화**. 단순 검색을 넘어 시맨틱(의미) 검색과 결합해 정확한 단어를 몰라도 원하는 정보를 찾아낸다.

- 답변 하단에 출처 노트 링크 자동 표시 → 신뢰성 확보
- ChatGPT처럼 대화하지만 **내 자료가 컨텍스트**라는 점에서 일반 AI 도구와 차별화

#### 3️⃣ 인사이트 분석 & 학습 공백 탐지 (7.1.1, 11.1)

단순 저장을 넘어, 쌓인 노트를 AI가 분석해 **새로운 인사이트 도출 + 지식 공백 영역 발견 + 추가 학습 추천**까지 제공한다.

- "내가 무엇을 많이 공부했고 어디가 부족한지"를 자동 리포트로 확인
- 지식 성장 대시보드(스트릭, TOP5 연결 노트 등)로 성장 과정 시각화
### 차별점 정리 (발표용 압축본)

기존 정리해주신 4가지 컨셉을 발표 시간에 맞춰 **핵심 1개 + 보조 2개**로 구조화하는 것을 제안한다.

#### 🎯 메인 차별점 (가장 강조할 것)

**"수동형 vs 자율형 세컨드 브레인"**

> Obsidian은 당신이 직접 연결해야 하는 도구, BrainX는 AI가 알아서 연결하는 자율 두뇌

|구분|Obsidian|BrainX|
|---|---|---|
|연결 방식|사용자가 직접 `[[ ]]`|AI가 자동 제안|
|탐색|폴더+수동 그래프|RAG 챗봇 대화|
|AI 활용|외부 플러그인 (복잡)|내장 + 원클릭 전환|

#### 🌟 보조 차별점 (시간 있으면 추가)

1. **시간의 흐름을 반영**: 정적 문서가 아니라 타임랩스 애니메이션 + 미열람 노드 흐리기로 "살아있는 지식"을 표현
2. **개방형 허브**: Notion·Obsidian·블로그와 심리스 동기화 → 기존 자료를 다 버리지 않고 가져올 수 있음

"Obsidian은 당신이 '정리'하게 만들지만, BrainX는 당신이 '생각'에만 집중하게 만듭니다."

- **기술 선정 이유 (Rationale):** 
	- ### 기술 스택 & 선정 이유

| 기술                          | 선정 이유                                                                                                                                                       |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Spring Boot (MSA)**       | 5개 서비스가 서로 다른 책임(인증, AI, 결제 등)을 가지므로, 모놀리식으로 묶으면 AI 모델 교체나 결제 모듈 점검 시 전체 서비스를 재배포해야 하는 리스크가 생긴다. 서비스별 독립 배포·스케일링을 위해 MSA를 채택했다.                             |
| **PostgreSQL**              | 노트, 사용자, 결제 등 **관계형 데이터의 정합성(트랜잭션)**이 중요한 도메인이 많아 RDBMS를 기본으로 선택했다.                                                                                         |
| **qdrant**                  | 별도의 벡터 DB(Pinecone 등)를 새로 운영하면 인프라가 하나 더 늘어나고 PostgreSQL과의 데이터 동기화 문제가 생긴다. **PostgreSQL 확장으로 벡터 검색까지 처리**해 인프라를 단순화하고, 노트 메타데이터와 임베딩을 같은 트랜잭션 안에서 다룰 수 있다. |
| **Redis (In-memory)**       | 최근 열람 노트, 세션, 인증 토큰처럼 **빈번하게 읽고 TTL이 필요한 데이터**를 캐싱해 DB 부하를 줄이고 응답 속도를 높인다.                                                                                  |
| **Kafka**                   | RAG 임베딩 생성, 토큰 사용량 집계처럼 **즉시 응답이 필요 없는 작업**을 비동기로 분리해 서비스 간 결합도를 낮췄다. Workspace-Service가 AI-Service의 처리 속도에 종속되지 않도록 하는 것이 핵심 목적이다.                         |
| **Docker / Kubernetes**     | 5개 서비스 + 여러 DB가 혼재된 환경에서 로컬·운영 환경 차이를 없애고(Docker), 서비스별 오토스케일링과 장애 시 자동 재시작(K8s)으로 가용성을 확보한다.                                                               |
| **AWS EC2**                 | 비용 효율적으로 컨테이너 오케스트레이션 환경을 구축할 수 있는 클라우드 인프라로 선정했다.                                                                                                          |
| **Async API / Open API 병행** | 사용자 응답이 즉시 필요한 CRUD는 **Open API(동기)**, AI 처리·통계 집계처럼 지연이 허용되는 작업은 **Async API(Kafka 기반)**로 구분해 트래픽 특성에 맞는 통신 방식을 사용했다.                                      |
- **데이터베이스 설계 (ERD) 또는 유즈케이스 (Usecase):** 
	- **1. AI 자동 연결(백링크 제안)**

	```
	노트 저장(Workspace) → Kafka 이벤트 → AI-Service가 pgVector에서 유사 노트 검색
	→ 유사도 점수 포함된 "연결 제안" 응답 → Workspace-Service가 사용자에게 표시
	→ 사용자 승인 시에만 실제 링크 생성(Workspace DB)
	```
	
	**2. RAG 기반 챗봇**
	
	```
	사용자 질문 → API Gateway → AI-Service
	→ pgVector에서 관련 노트 청크 검색(유사도 기반)
	→ 검색된 컨텍스트 + 질문을 LLM에 주입 → 답변 생성
	→ 출처 노트 ID 포함해 응답 → 클릭 시 Workspace-Service로 이동
	```
	
	**3. 인사이트/학습 공백 분석**
	
	```
	사용자 요청(분석 리포트 생성) → AI-Service가 Workspace-Service의 전체 노트 메타데이터 조회
	→ 임베딩 클러스터링으로 주제 분포 분석 → 공백 영역 도출
	→ 리포트 생성 완료 → Kafka 이벤트로 User-Service에 알림(푸시/이메일) 트리거
	```

- 이 구조에서 강조할 포인트는, **AI-Service가 다른 서비스의 DB를 직접 건드리지 않고 API/이벤트로만 데이터를 주고받는다**는 것이다. 이를 통해 서비스 간 결합도를 낮추고, AI 모델이나 벡터 DB를 교체해도 다른 서비스에 영향이 없다는 MSA의 핵심 장점이 드러난다.
# 역할분담
**김채영**
- 팀장, Backend PM -> Backend 구조 설계, MSA 설계 및 연동
    - **User-Service** (PostgreSQL, Redis(In-memory)) — 사용자 신원, 인증(로그인/회원가입/온보딩), 계정 보안, 동의, 마이페이지, 노트 사용 통계
    - **Admin-Service** (PostgreSQL) — 관리자 페이지, 사용자 관리, 결제 관리(환불 처리 등), 모니터링, 사용자 통계, 문의 답장, 사용자/모델별 LLM 토큰 소비량

**고영진**
- DevOps PM -> CI/CD 파이프라인, RAG 구현, LLM 정제
    - **AI-Service** (PostgreSQL, VectorDB) — 시맨틱 검색, RAG, LLM, AI 추천, 요약, 토큰 사용량(서비스 단위)
    - CI/CD 파이프라인 구축 및 운영 (GitHub Actions, EC2 배포)

**진환유**
- AI PM -> 외부 연동 및 결제 시스템 설계
    - **Ingestion-Service** (PostgreSQL) — 파일 처리, 변환, 가져오기(Notion/Obsidian), 내보내기, 외부 연동
    - **Commerce-Service** (PostgreSQL) — 결제(결제 API), 플랜 관리

**이예진**
- Git PM -> Git 전략 수립, 브랜치 관리, 노트 도메인 설계

    - Git 협업 전략 수립 (브랜치 정책, 머지 컨플릭트 관리, monorepo 운영)

**이진주**
- Frontend PM -> Frontend 공통 UI 개발, 그래프 시각화
    - **Workspace-Service** (PostgreSQL, GraphDB, Redis(In-memory)) — 그래프(마인드맵 노드 시각화)
    - Frontend 공통 UI 개발 (Next.js 기반 디자인 시스템, 공통 컴포넌트)

|검색·발견|정확한 키워드를 기억해야 검색 가능|시맨틱 검색으로 의미 기반 탐색, 못 찾던 노트도 발견|
|정리 부담|폴더링·태깅을 전부 사용자가 수행|AI 자동 연결 제안 + 폴더 정리 제안으로 정리 시간 절감|
|지식 활용|단순 저장 후 방치(휘발성 기억)|RAG 챗봇 대화로 과거 기록을 즉시 활용, 인사이트 리포트로 학습 공백 파악|
|생태계 이전|Notion/Obsidian 자료가 고립됨|가져오기/내보내기로 기존 자료를 손실 없이 통합|

**한 줄 요약**: 사용자는 "기록하는 행위"에만 집중하고, 정리·연결·발견은 AI가 대신 수행 → **지식 관리에 들던 시간이 실제 생산적 사고(학습, 집필, 의사결정)로 전환**된다.

#### 비즈니스적 가치

**1. 수익 모델 (Freemium → 구독)**

- 무료 플랜: 기본 노트 작성 + 키워드 검색
- Pro/Team 플랜: 시맨틱 검색, RAG 챗봇, AI 클러스터링 등 토큰 기반 과금 기능 — 사용량에 비례한 안정적 매출 구조 확보
- 결제 연동(Stripe/토스페이먼츠)으로 구독 업/다운그레이드, 환불까지 자동화 → 운영 비용 절감

**2. 데이터 락인 효과(Lock-in)**

- 사용자가 쌓은 노트와 AI가 학습한 연결 구조(마인드맵)는 시간이 지날수록 더 정교해짐 → 타 서비스로의 이전 비용이 커지며 자연스러운 재구독 유도

**3. 시장 포지셔닝**

- "AI 도구"와 "노트 도구" 사이의 빈 시장을 공략 → 두 카테고리 사용자 모두를 잠재 고객으로 확보 가능 (ChatGPT 사용자 + Notion/Obsidian 사용자)

**4. 확장 가능성**

- MCP 서버를 통한 외부 AI 에이전트 연동 → B2B 영역(기업 내부 지식관리, 사내 챗봇)으로 확장 가능
- 크롬 확장, 블로그 템플릿 등 부가 기능으로 자연스러운 트래픽 유입 채널 확보

#### 기술적 리스크

|리스크|발생 가능 영역|대응 방안 (Plan B)|
|---|---|---|
|**벡터 검색 응답 지연**|AI-Service, pgVector|청크 사이즈/overlap 최적화 우선 적용 → 임계치 초과 시 별도 벡터 DB(Pinecone 등) 분리 검토|
|**LLM 비용 급증**|AI-Service, 토큰 과금|사용자별 토큰 사용량 모니터링(Admin-Service) 선제 구축, 캐싱으로 중복 질의 비용 절감, 무료 플랜은 자체 모델/제한된 토큰으로 운영|
|**MSA 간 데이터 정합성 깨짐**|서비스 간 Kafka 이벤트|이벤트 실패 시 재시도(retry) + Dead Letter Queue 적용, 최종적 일관성(eventual consistency) 전제로 설계|
|**외부 API 의존성** (Notion, OAuth 등)|Ingestion-Service|Notion API rate limit(3req/s) 고려해 큐 기반 처리, 실패 시 부분 가져오기 + 재시도 안내|
|**파일 형식 변환 실패** (hwpx 등)|Ingestion-Service|1차는 오픈소스 라이브러리로 대응, 변환 실패 시 원본 첨부파일로 유지(텍스트 변환 강제하지 않음)|
|**그래프 시각화 성능 저하** (노드 1,000개 이상)|Workspace-Service, Front|캔버스 최적화 우선 적용, 기기 사양별 품질 옵션(low/mid/high)으로 단계적 대응|

#### 일정 리스크

|리스크|대응 방안|
|---|---|
|**우선순위 미준수로 일정 지연**|P0(출시 필수) 기능만 1차 마일스톤으로 설정, P1~P3는 단계적 출시(MVP → 정식 버전)로 분리|
|**5인 팀의 서비스 간 의존성** (예: Workspace 완료 전까지 AI-Service 테스트 불가)|API 명세(OpenAPI) 우선 확정 → Mock 서버 기반 병렬 개발로 의존성 최소화|
|**신규 기술 학습 곡선** (Kafka, K8s 등 팀 전체가 처음 다루는 스택)|학습 전용 스프린트 별도 배정, 초기엔 단순화된 구조(Kafka 토픽 최소화 등)로 시작 후 점진적 확장|
|**AI 기능 품질 불확실성** (연결 제안, 클러스터링 정확도)|사용자 승인 기반 설계(자동 미실행)로 AI 오류의 사용자 영향 최소화, 베타 테스트로 정확도 검증 후 정식 적용|

#### 핵심 원칙

> 모든 AI 자동화 기능은 **사용자 승인 없이는 실행되지 않는 구조**(연결 생성, 폴더 재편성 등)로 설계되어, AI의 판단 오류가 발생해도 사용자 데이터에 직접적인 손상을 주지 않는다. 이는 기술적 한계를 비즈니스 리스크로 확산시키지 않는 안전장치 역할을 한다.
