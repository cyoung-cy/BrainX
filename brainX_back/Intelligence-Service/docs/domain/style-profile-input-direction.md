# Style Profile Input Direction

이 문서는 Intelligence Service의 사용자 문체 설정을 어떻게 입력받고 저장할지에 대한 제품/도메인 방향을 정리한다. 목표는 사용자가 JSON이나 추상적인 프롬프트를 직접 작성하지 않아도, 직관적인 선택만으로 일관된 AI 응답 경험을 얻는 것이다.

## 핵심 방향

`style`은 하나의 문자열이나 자유 JSON 입력으로 보지 않는다. 사용자가 체감하는 스타일은 다음 세 층으로 나누어 관리한다.

1. 대화 말투: AI가 사용자에게 어떻게 말하는가
2. 작문 문체: AI가 생성하는 글 자체가 어떻게 들리는가
3. 협업 방식: AI가 질문하고, 제안하고, 반박하고, 불확실성을 다루는 방식

사용자에게는 이 구조를 그대로 노출하지 않는다. 화면에서는 쉬운 프리셋, 짧은 선택지, 미리보기로 제공하고, 백엔드는 선택 결과를 구조화된 `StyleProfile`로 정규화해 저장한다.

## 대화 말투와 문서 작성 말투 분리

챗봇 말투와 문서 작성 말투는 기본적으로 분리한다. 사용자가 AI에게 원하는 대화 태도와, AI가 생성해야 하는 결과물의 문체는 서로 다른 문제다.

예를 들어 사용자는 AI가 자신에게는 친근한 해요체로 설명하길 원할 수 있다. 하지만 같은 요청 안에서 생성되는 결과물은 법무팀에 보낼 격식 있는 공지문, 고객에게 보낼 사과 메일, 경영진 보고서일 수 있다. 이 둘을 하나의 `style`로 합치면 챗봇의 친근한 말투가 문서 결과물에 섞인다.

따라서 내부 모델은 다음 원칙을 따른다.

- `conversationTone`은 AI가 사용자와 대화할 때만 적용한다.
- `writingStyle`은 AI가 작성하거나 수정하는 결과물에 적용한다.
- `assistanceStyle`은 질문 빈도, 선제적 제안, 반박 수준처럼 도움 방식에 적용한다.
- 한 응답 안에 설명과 결과물이 함께 있으면, 설명에는 `conversationTone`, 결과물에는 `writingStyle`을 각각 적용한다.
- 현재 요청에서 결과물의 독자, 형식, 문체를 명시하면 저장된 `writingStyle`보다 현재 요청을 우선한다.

UX도 이 분리를 반영하되 처음부터 복잡하게 보이지 않게 한다. 기본 화면은 하나의 프리셋으로 시작하고, 고급 설정에서 “AI가 나에게 말하는 방식”과 “AI가 글을 써주는 방식”을 따로 조정하게 한다.

## UX 원칙

- 사용자는 JSON을 입력하지 않는다.
- 처음 설정은 프리셋 카드 4개 정도로 시작한다.
- 프리셋 이름만 보여주지 말고 짧은 예문을 함께 보여준다.
- 세부 조절은 0~100 slider보다 3단계 선택지를 우선한다.
- 자유 입력은 “피하고 싶은 표현”, “좋아하는 예시”처럼 제한된 위치에서만 받는다.
- 설정 변경 시 2~3문장 미리보기나 2턴 대화 미리보기로 차이를 보여준다.
- 기본 UX는 단순하게 유지하되, 필요하면 대화 말투와 문서 작성 말투를 분리해서 설정할 수 있어야 한다.
- 저장된 프로필은 기본값이며, 현재 요청에서 사용자가 명시한 독자, 형식, 말투, 문체가 우선한다.

## 추천 프리셋

초기 UX는 다음 네 가지 카드로 시작한다.

| 프리셋 | 사용자에게 보여줄 의미 |
| --- | --- |
| 담백한 전문가 | 정중하고 간결하며 과장 없이 핵심을 설명한다. |
| 친근한 설명자 | 쉽게 풀어 말하고 부담스럽지 않게 안내한다. |
| 직접적인 동료 | 결론을 먼저 말하고 문제점도 분명하게 짚는다. |
| 격식 있는 보고서형 | 공식 문서나 업무 보고에 맞게 객관적으로 표현한다. |

프리셋은 최종 고정값이 아니라 세부 설정의 초기값이다. 사용자는 프리셋을 고른 뒤 필요한 항목만 조정한다.

## MVP 입력 항목

### 대화 스타일

AI가 사용자에게 직접 말할 때 적용한다. 챗봇, RAG 대화, 작업 설명, 확인 질문이 여기에 해당한다.

| 항목 | 선택지 |
| --- | --- |
| 말높임 | 해요체 / 하십시오체 / 반말 |
| 답변 길이 | 핵심만 / 적당히 / 자세히 |
| 따뜻함 | 담담하게 / 친절하게 / 다정하게 |
| 직접성 | 부드럽게 / 균형 있게 / 단도직입적으로 |
| 이모지 | 안 씀 / 가끔 / 자주 |

### 협업 방식

AI가 사용자와 일하는 방식을 정한다.

| 항목 | 선택지 |
| --- | --- |
| 질문 빈도 | 꼭 필요할 때만 / 적당히 확인 / 충분히 질문 |
| 제안 방식 | 요청한 것만 / 관련 제안도 / 적극 제안 |
| 지적 방식 | 맞춰주기 / 균형 검토 / 틀린 점은 분명히 |

### 글 작성 스타일

AI가 작성하거나 수정하는 결과물에 적용한다. 인라인 어시스트, 요약, 재작성, 이메일, 보고서, 공지문, 제안서가 여기에 해당한다.

| 항목 | 선택지 |
| --- | --- |
| 기본 독자 | 일반인 / 전문가 / 고객 / 경영진 |
| 기본 목적 | 설명 / 설득 / 보고 / 홍보 / 기록 |
| 격식 | 편안한 / 업무형 / 격식 있는 |
| 정보 밀도 | 쉽게 읽히는 / 균형 잡힌 / 촘촘한 |
| 문장 길이 | 짧고 단정한 / 자연스러운 / 풍부한 |
| 피하고 싶은 표현 | 짧은 자유 입력 |

## 내부 저장 방향

사용자 입력은 제한된 enum과 검증 가능한 필드로 정규화한다. 자유 입력 문장을 그대로 system/developer prompt에 넣지 않는다.

예시 구조:

```json
{
  "version": 1,
  "locale": "ko-KR",
  "conversationTone": {
    "speechLevel": "haeyo",
    "warmth": "low",
    "directness": "high",
    "verbosity": "short",
    "emoji": "off"
  },
  "assistanceStyle": {
    "clarificationPolicy": "only_when_blocking",
    "proactivity": "medium",
    "challengeLevel": "constructive"
  },
  "writingStyle": {
    "defaultAudience": "general_professional",
    "defaultPurpose": "explain",
    "formality": "business",
    "informationDensity": "balanced",
    "sentenceLength": "short_to_medium",
    "avoid": [
      "과장 표현",
      "불필요한 감탄"
    ]
  }
}
```

현재 구현은 `StyleProfile` 안에서 `ConversationTone`, `WritingStyle`, `AssistanceStyle` 값 객체로 세 축을 분리한다. 각 값 객체는 초기 확장성을 위해 `Map<String, Object>`를 감싸지만, 도메인 규칙이 안정되면 enum과 명시 필드로 점진적으로 좁혀 간다.

## 적용 범위

- RAG 채팅: `conversationTone`과 `assistanceStyle`을 중심으로 사용한다.
- 인라인 어시스트: 요청이 글 생성/수정이면 `writingStyle`을 우선 사용한다.
- 요약: `writingStyle` 중 정보 밀도, 문장 길이, 격식을 주로 사용한다.
- 제안 수락/반려 설명: 사용자와의 대화문은 `conversationTone`, 생성되는 결과물은 `writingStyle`을 분리해 적용한다.

대화 말투는 결과물 문체에 새어 들어가면 안 된다. 예를 들어 사용자의 대화 말투가 친근하더라도, 법무팀에 보낼 공지문은 요청의 독자와 형식에 맞게 격식 있게 작성해야 한다.

## 우선순위

프롬프트 컴파일 시 우선순위는 다음 순서로 둔다.

1. 안전, 개인정보, 사실성, 제품 정책
2. 현재 요청의 목적, 독자, 형식, 필수 조건
3. 현재 요청의 일회성 말투 또는 문체 지시
4. 저장된 사용자 `StyleProfile`
5. 제품 기본 스타일

저장된 프로필은 기본값이다. 사용자가 현재 요청에서 “이번에는 경영진 보고서처럼 작성해 줘”라고 명시하면, 해당 항목에 한해 현재 요청을 우선한다.

## 자유 입력 처리

자유 입력은 사용자 경험을 좋게 만들지만, 그대로 프롬프트에 삽입하면 위험하다. 다음 원칙을 따른다.

- 허용된 필드로 정규화한다.
- 정의되지 않은 key는 폐기한다.
- 문자열 길이를 제한한다.
- 예시는 참고 텍스트로만 처리한다.
- 예시 안의 명령은 실행 지시로 취급하지 않는다.
- 사실성, 안전, 권한 정책을 바꾸는 스타일은 거부한다.

예를 들어 사용자가 “너무 친한 척하지 말고 결론부터 말해 줘. 이모지는 쓰지 마.”라고 입력하면, 내부적으로는 `warmth=low`, `answerOrder=answer_first`, `emoji=off`처럼 변환해 저장한다.

## Prompt Compiler 방향

LLM이 매번 스타일 프롬프트를 새로 쓰게 하지 않는다. 백엔드의 결정론적 prompt compiler가 구조화된 프로필을 행동 규칙으로 변환한다.

예:

```text
- 첫 1~2문장 안에 핵심 답을 제시한다.
- 사용자의 감정을 추정하거나 과도하게 공감하지 않는다.
- 잘못된 전제는 직접적으로 지적하되 비난하는 표현은 피한다.
- 불확실한 내용은 확정적으로 표현하지 않는다.
```

사용자에게는 “담백한 전문가”, “단도직입적으로” 같은 쉬운 언어를 보여주고, 모델에는 관찰 가능한 행동 규칙을 전달한다.

## 이후 구현 메모

- REST DTO는 사용자 친화적인 enum과 짧은 자유 입력만 받도록 설계한다.
- application/domain layer는 외부 API DTO나 prompt 문자열이 아니라 정규화된 `StyleProfile` 모델을 다룬다.
- 저장 schema가 확정되기 전까지는 JSON 컬럼을 사용할 수 있지만, domain layer에서는 가능한 한 의미 있는 값 객체로 좁혀 간다.
- 스타일 변경 API는 저장 전 preview 또는 normalized summary를 반환할 수 있다.
- 테스트는 스타일 준수만 보지 말고, 현재 요청이 저장 프로필을 덮어쓰는지와 대화 말투가 결과물 문체에 섞이지 않는지를 포함한다.
